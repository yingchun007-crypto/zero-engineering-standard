本文件用于约束零维团队项目的测试补充、测试审查和测试风险说明。具体命令和工具以项目现有配置为准。

## 适用范围

- 新增功能测试。
- Bug 修复回归测试。
- 接口、Service、Repository / Mapper、组件、页面和工具函数测试。
- 数据库迁移、权限、安全、第三方调用和异步任务测试。
- 无法补测试时的风险说明。

## 基本原则

- 测试优先覆盖业务风险，而不是追求无意义覆盖率数字。
- Bug 修复必须优先补回归测试，除非项目没有测试框架或成本明显不合理。
- 测试应稳定、可重复、可独立运行。
- 测试不依赖本地脏数据、真实第三方服务或真实生产配置。
- 时间、随机数、并发、外部 IO 必须可控。

## 测试分层

- 单元测试：验证纯业务规则、工具函数、转换逻辑和边界条件。
- Service 测试：验证业务流程、事务边界、异常分支和权限判断。
- API / Controller / Router 测试：验证入参校验、响应结构、错误码和鉴权。
- Repository / Mapper 测试：验证复杂查询、分页、排序、索引相关条件和 SQL 映射。
- 前端组件 / 页面测试：验证表单、表格、交互、状态和权限显示。
- 集成测试：验证多个模块、数据库、缓存、消息或第三方适配的协作。

## 必须补测试的场景

- 修复线上或明确复现的 bug。
- 新增核心业务流程。
- 修改权限、认证、数据归属或安全规则。
- 修改统一响应、全局异常、请求封装或路由守卫。
- 修改数据库复杂查询、分页、排序或迁移逻辑。
- 修改金额、时间、状态机、幂等和批量操作。
- 修改第三方支付、回调、消息消费、导入导出等高风险流程。

## 测试数据

- 测试数据应独立构造，不依赖开发者本机数据库状态。
- 测试数据应表达业务语义，避免只有 `test1`、`foo`、`bar`。
- 敏感信息必须使用假数据。
- 数据库测试应能清理数据或使用事务回滚、临时库、容器库等隔离策略。
- 初始化数据应尽量幂等，避免重复运行失败。

## Mock 与外部依赖

- 单元测试不调用真实第三方服务。
- 第三方 HTTP、短信、支付、对象存储、消息队列应使用 Mock、Stub、Fake 或测试环境替身。
- Mock 应验证关键入参和异常分支，不只让测试“跑绿”。
- 对外部服务超时、失败、空响应、非 2xx、业务失败码应有测试或明确风险说明。

## 断言要求

- 测试必须断言业务结果、错误码、关键字段和副作用。
- 异常场景应断言异常类型、错误码或用户可理解错误信息。
- 写操作应断言数据库、状态、事件、日志或外部调用等关键副作用。
- 不使用只验证“不报错”的弱断言替代业务断言。

## 前端测试

- 表单必须覆盖必填、格式、边界和提交失败。
- 表格页面应覆盖查询、重置、分页、加载态和空状态。
- 权限相关页面应覆盖无权限、只读、按钮隐藏或禁用。
- 请求错误、token 过期、业务失败应遵循统一 request 处理，不在页面重复造逻辑。
- 如项目未配置前端测试框架，应至少执行 `typecheck`、`lint`，并说明人工验证点。

## 后端测试

- 参数校验应覆盖空值、非法格式、越界和枚举非法值。
- Service 测试应覆盖成功、失败、边界和事务回滚。
- API 测试应覆盖响应结构、错误码、鉴权和权限不足。
- Repository / Mapper 测试应覆盖复杂 SQL、逻辑删除、分页和排序。
- 涉及时间的测试应固定时钟或显式构造时间。

## 测试交付清单

- 是否补充或更新了与变更风险匹配的测试。
- Bug 修复是否覆盖复现路径和回归断言。
- 核心业务是否覆盖成功、失败、边界场景。
- 权限、安全、兼容性、事务和数据一致性是否有测试或风险说明。
- 是否避免真实第三方服务和真实生产敏感配置。
- 是否执行项目对应测试命令，未执行时是否说明原因。
