## 定位

本文件约束前端和后端代码注释。具体语法由对应技术栈规范补充：Java 使用 Javadoc，Python 使用 docstring，TypeScript / Vue 使用 TSDoc 或简短块注释。

## 通用原则

- 注释说明业务意图、输入输出语义、边界条件、权限约束、异常语义和不明显的实现原因。
- 注释与代码必须同步更新；修改行为时同步修改类、方法、参数、返回值和字段说明。
- 公共 API、Service 接口、跨模块类型、复杂工具函数必须让调用方不读实现也能理解如何使用。
- 简单私有方法可不写注释，但方法名和参数名必须自解释。
- 业务注释优先使用中文；与框架、协议、库名相关的固定术语保持英文。

## 必须写注释的场景

- 对外接口、Controller / Router、Service 接口、Repository / Mapper 复杂查询方法。
- 公共类、公共函数、公共类型、公共常量、公共枚举和错误码。
- DTO / Schema / VO / TypeScript 接口中的关键字段、枚举字段、状态字段、金额、时间、ID、权限字段。
- 复杂业务流程、状态流转、幂等、防重、权限判断、数据归属、事务边界和补偿逻辑。
- 与安全、脱敏、加密、签名、回调、第三方接口、文件、导入导出、批量操作相关的代码。
- 临时兼容、历史包袱、特殊分支和看似多余但不能删除的逻辑。

## 命名与注释配合

- 类名、方法名、变量名必须先清晰表达含义，不能依赖注释解释含糊命名。
- 方法注释必须写清方法用途、关键参数语义、返回值语义和主要异常 / 错误码。
- 布尔参数、状态参数、类型参数必须在注释或类型中说明取值含义。
- 返回集合时说明空集合、分页、排序和过滤语义；返回 `null` / `None` 时说明业务含义。
- 有副作用的方法必须说明写入范围、事务边界、事件、消息、外部调用或缓存影响。

## 禁止事项

- 不写重复代码本身的注释，如“设置用户名”“循环列表”。
- 不保留大段注释掉的废弃代码；需要保留原因时写清原因、负责人或删除条件。
- 不在注释中写密码、Token、密钥、真实手机号、身份证、生产地址或内部漏洞细节。
- 不用错误注释掩盖糟糕命名；优先改名、拆分方法或抽取类型。
- 不为所有 getter、setter、简单构造器和显而易见的局部变量机械补注释。

## 交付清单

- 公共类、接口、方法、参数、返回值是否有必要注释。
- 复杂业务、权限、安全、事务、异常、兼容逻辑是否说明原因和边界。
- DTO / VO / Schema / TypeScript 类型字段是否有说明。
- 注释是否与实现一致，是否存在过期、误导或重复代码的注释。
- 是否避免在注释中暴露敏感信息。
