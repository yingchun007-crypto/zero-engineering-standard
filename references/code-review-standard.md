本文件用于约束零维团队代码审查、PR / MR 审查和 AI Review。Review 的核心目标是让代码库长期更健康，而不是追求完美、偏好争论或机械挑刺。

## 目标

- 发现真实缺陷、数据风险、安全风险、兼容性风险和测试缺口。
- 保持架构、接口、权限、数据库、依赖和代码风格与项目一致。
- 帮助作者尽快交付可维护的改进。
- 通过清晰反馈沉淀团队知识，而不是制造无谓返工。

## 审查标准

- 当变更整体提升代码健康且不存在阻塞问题时，应倾向通过；不要因非关键瑕疵长期阻塞。
- 技术事实、数据、项目规范和现有架构优先于个人偏好。
- 纯风格问题只有在违反项目配置、团队规范或明显降低可读性时才应阻塞。
- 设计问题、功能错误、安全漏洞、数据不一致、接口破坏、迁移风险和缺失关键测试可以阻塞。
- 如果有多个合理实现，作者能基于项目上下文说明理由时，应尊重作者方案。

## 审查顺序

1. 先读需求、PR / MR 描述、关联 issue 和测试说明，判断变更是否应该发生。
2. 先看核心文件和主要设计，再看次要文件；重大设计问题应尽早反馈。
3. 必要时先读测试，理解预期行为和边界。
4. 对被要求审查的人写代码逐行理解；生成文件、大数据文件可按风险抽查。
5. 如果只审查了部分范围，必须说明审查范围和未覆盖部分。

## 输出格式

Review 输出必须先列问题，再给总结。发现问题时使用：

```text
[P1] 标题
文件:行号
问题说明、触发条件、影响范围和建议修正。
```

没有发现问题时，明确说明未发现阻塞性问题，并列出剩余风险、未执行检查或未覆盖测试。

行内评论可使用更轻的格式：

```text
issue (blocking): 这里会在未登录时绕过权限校验。
suggestion (non-blocking): 可以复用现有转换函数，减少重复映射。
question: 这个状态是否可能被历史数据置为 null？
```

## 严重级别

- `P0`：严重安全事故、数据破坏、生产不可用、不可逆损失。
- `P1`：功能错误、越权、数据不一致、接口破坏性变更、高概率线上故障。
- `P2`：边界错误、测试缺口、性能隐患、兼容性风险、维护成本明显上升。
- `P3`：命名、局部可读性、低风险风格或可选改进。

默认只报告 `P0`、`P1`、`P2`。`P3` 仅在用户要求细致风格审查，或它能明显降低后续维护成本时报告。

## 评论标签

- `issue`：确认存在的问题；阻塞时标记 `(blocking)`。
- `suggestion`：明确改进建议；非必须时标记 `(non-blocking)`。
- `question`：需要作者澄清的潜在风险，不确定时不要伪装成结论。
- `nitpick`：非常小的偏好或风格问题，默认非阻塞。
- `note`：补充背景或提醒，默认非阻塞。
- `praise`：真实指出做得好的地方，可用于团队知识传递，但不要虚假表扬。

评论应解释“为什么”，并尽量给出触发条件、影响和修正方向。评论针对代码，不针对作者。

## 重点清单

通用：

- 是否满足需求，是否遗漏关键场景、边界值、失败路径或并发场景。
- 是否引入无关重构、无关格式化、范围膨胀或删除既有业务逻辑。
- 命名、注释和文档是否帮助后续维护；注释是否解释原因而不是重复代码。

API：

- 是否破坏请求 / 响应结构、字段类型、必填规则、错误码或枚举语义。
- 分页、排序、过滤、幂等、批量和文件接口是否遵守 API 规范。

安全：

- 是否校验认证、鉴权、数据归属和敏感信息脱敏。
- 是否存在注入、路径穿越、SSRF、XSS、CSRF、日志泄露或不可信回调。

数据库：

- 是否有迁移脚本、回滚方案、索引策略、历史数据处理和锁表风险评估。
- 是否避免无条件全表更新 / 删除、大事务长时间持锁和逻辑删除唯一约束风险。

后端：

- Controller / Router 是否过重。
- 事务、异常、日志、DTO / VO 转换、第三方调用超时和重试边界是否清晰。

前端：

- 是否复用 request、router、store、hooks、i18n、权限模型和设计系统。
- 页面状态、表单校验、错误提示、加载态、空状态和权限态是否完整。

测试：

- Bug 修复是否有回归测试。
- 核心业务是否覆盖成功、失败、边界和权限场景。
- 测试是否真的会在代码坏掉时失败，是否避免弱断言和过度复杂测试。

## 专家审查触发

出现以下变更时，应请求对应领域专家或维护者审查，或在 Review 中明确提示：

- 安全、认证、鉴权、租户、权限策略、敏感数据。
- 数据库迁移、复杂查询、大表 DDL、索引和性能风险。
- 新增依赖、License 不确定、包体积明显增加、基础设施变更。
- 用户可见 UI / UX、可访问性、国际化和关键业务流程。
- 支付、订单、消息、导入导出、异步任务、第三方回调等高风险流程。

## AI Review 约束

- 必须基于具体 diff、文件和行号提出问题；没有证据时用 `question` 或说明假设。
- 不臆造项目不存在的规范、接口、字段或测试命令。
- 不把“缺少我偏好的重构”当成阻塞问题。
- 不要求作者处理与本次变更无关的历史问题；可作为 `note` 或后续建议。
- 对重大问题优先给最小修复路径；对非阻塞建议明确标记。
- 如果上下文不足以判断，说明需要补充的文件、运行结果或业务规则。
