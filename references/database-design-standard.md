本文件用于约束项目数据库表结构、字段类型、索引、约束、SQL 变更和数据安全规范。后端 Entity、MyBatis-Plus、分页和事务规范详见 `references/springboot-development-standard.md` 或对应后端开发规范。

## 适用范围

- 业务表设计
- 字段类型设计
- 主键、唯一键、索引设计
- 逻辑删除和审计字段
- SQL 脚本和数据库变更
- 查询性能和数据安全

如项目已使用 Flyway、Liquibase 或公司级数据库规范，应以更高优先级规范为准。

## 基本原则

- 表结构必须服务业务语义，避免为了短期开发便利牺牲长期可维护性。
- 表名、字段名必须稳定、清晰、可读。
- 字段类型必须与业务含义匹配，避免使用过大、过宽或含糊类型。
- 所有业务表必须有主键。
- 重要业务唯一性必须由数据库唯一约束兜底。
- 频繁查询条件必须有合理索引。
- 禁止无条件全表更新和删除。
- 数据库变更必须可追踪、可回滚、可评审。

## 命名规范

通用要求：

- 表名、字段名、索引名全部使用小写下划线。
- 名称应表达业务含义，禁止使用拼音、无意义缩写和模糊命名。
- 禁止使用数据库关键字，如 `order`、`group`、`user` 等。
- 同一业务概念在不同表中命名必须一致。

表名：

- 业务表建议使用模块前缀，如 `sys_user`、`sys_role`、`biz_file`。
- 关系表建议使用两个实体名组合，如 `sys_user_role`。
- 日志表建议使用 `_log` 后缀。
- 配置表建议使用 `_config` 后缀。
- 明细表建议使用 `_detail` 后缀。

字段名：

- 主键统一使用 `id`。
- 外键字段使用 `{业务名}_id`，如 `user_id`、`role_id`。
- 状态字段使用 `status` 或明确业务前缀，如 `audit_status`。
- 类型字段使用 `type` 或明确业务前缀，如 `file_type`。
- 时间字段使用 `_time` 后缀，如 `create_time`、`expire_time`。
- 数量字段使用 `_count` 后缀。
- 金额字段使用 `_amount` 后缀。
- 布尔语义字段使用 `is_`、`has_`、`enable_` 等清晰前缀。

索引名：

```text
主键索引：pk_表名
唯一索引：uk_表名_字段名
普通索引：idx_表名_字段名
联合索引：idx_表名_字段1_字段2
```

示例：

```text
pk_sys_user
uk_sys_user_username
idx_sys_user_mobile
idx_sys_user_status_create_time
```

## 基础字段规范

业务表默认包含：

```text
id            主键
create_time   创建时间
update_time   更新时间
create_by     创建人 ID
update_by     更新人 ID
deleted       逻辑删除标识，0-未删除，1-已删除
```

要求：

- 后端 Entity 应继承 `BaseEntity`。
- 业务表应尽量保持基础字段一致。
- 历史表或特殊表如无法包含完整审计字段，需在表设计评审中说明原因。
- `create_time` 只在创建时写入。
- `update_time` 每次更新时刷新。
- `deleted` 默认值为 `0`。

## 主键规范

- 主键字段统一命名为 `id`。
- Java 类型默认使用 `Long`。
- 数据库类型根据数据库选型使用 `bigint` 或等价类型。
- 主键必须无业务含义。
- 不建议使用手机号、编码、邮箱等业务字段作为主键。
- 分布式 ID、雪花 ID、自增 ID 等方案应全项目统一。

## 字段类型规范

字符串：

- 短文本使用 `varchar`。
- 长文本使用 `text`，但必须评估查询和索引影响。
- 有明确长度的字段必须设置合理长度。
- 手机号建议 `varchar(20)`。
- 邮箱建议 `varchar(128)`。
- 编码、编号类字段根据业务长度设置，避免无脑 `varchar(255)`。

数字：

- ID 使用 `bigint`。
- 状态、类型、枚举值可使用 `tinyint`、`smallint` 或 `varchar`，但同项目应统一。
- 计数字段根据范围选择 `int` 或 `bigint`。
- 金额必须使用 `decimal`，禁止使用浮点类型。
- 金额精度按业务定义，常见为 `decimal(18,2)`。

时间：

- 时间字段推荐使用 `datetime`。
- 日期字段使用 `date`。
- 不建议用字符串存储时间。
- 时间字段命名必须明确业务含义，如 `start_time`、`end_time`、`expire_time`。

布尔：

- 布尔语义字段建议使用 `tinyint`。
- 必须明确值含义，如 `0-否，1-是`。

JSON：

- JSON 字段仅用于扩展属性、第三方原始报文快照等场景。
- 禁止把核心查询条件长期放在 JSON 字段中。
- JSON 字段必须说明结构和使用场景。

## NULL 与默认值规范

- 能确定必填的字段必须 `NOT NULL`。
- 必须设置合理默认值，避免业务代码大量处理 null。
- 字符串字段默认值可根据业务使用空串，但敏感或语义明确字段不应滥用空串。
- 数值字段默认值必须有业务含义。
- 时间字段不建议使用无意义默认时间，如 `1970-01-01`。
- 可空字段必须在注释中说明为空时的业务含义。

## 枚举与状态字段规范

- 状态、类型、来源、渠道等有限值字段必须有明确值域。
- 数据库存储稳定 `code`，不要存储展示文案。
- 字段注释必须说明枚举值含义。
- 枚举值发布后禁止随意修改含义。
- 删除枚举值必须评估历史数据。

示例：

```sql
status tinyint not null default 1 comment '状态：1-启用，0-禁用'
```

## 索引设计规范

必须建索引的场景：

- 主键。
- 唯一业务键。
- 高频查询条件。
- 高频排序字段。
- 高频 join 字段。
- 外键语义字段，如 `user_id`、`role_id`。

联合索引要求：

- 遵循最左前缀原则。
- 高区分度字段优先。
- 等值查询字段优先于范围查询字段。
- 排序字段应结合查询条件一起设计。
- 联合索引字段不宜过多，通常不超过 5 个。

禁止或谨慎：

- 禁止为每个字段都建索引。
- 低区分度字段单独建索引需谨慎，如性别、布尔值。
- 大文本字段不直接建普通索引。
- 频繁更新字段建索引需评估写入成本。
- 模糊查询 `%keyword` 无法有效利用普通索引，应评估搜索方案。

## 唯一约束规范

重要业务唯一性必须使用数据库唯一索引保证。

常见场景：

- 用户名唯一。
- 手机号唯一。
- 业务编码唯一。
- 第三方流水号唯一。
- 幂等请求号唯一。

逻辑删除场景：

- 唯一索引设计必须考虑 `deleted` 字段。
- 如果允许软删后重新创建同名数据，唯一索引不能简单只使用业务字段；可使用删除时间、删除批次、归档表、部分唯一索引等方案，具体以数据库能力和业务要求为准。
- 仅使用 `deleted` 的 0/1 标识参与唯一索引时，要评估多次删除同一业务键是否会再次冲突。
- 如果历史数据不允许重复，应保持业务字段唯一。

## 关联关系规范

- 业务关联优先使用明确的 ID 字段。
- 关联字段命名使用 `{实体名}_id`。
- 是否使用数据库外键由项目统一决定。
- 即使不使用物理外键，也必须在业务和索引层保证数据一致性。
- 多对多关系必须使用关系表，不使用逗号分隔字符串保存多个 ID。

关系表示例：

```text
sys_user_role
id
user_id
role_id
create_time
update_time
create_by
update_by
deleted
```

## 字段注释规范

- 表必须有注释。
- 字段必须有注释。
- 状态、类型、枚举字段必须说明每个值含义。
- 金额字段必须说明单位。
- 时间字段必须说明业务含义。
- 可空字段应说明为空含义。

示例：

```sql
amount decimal(18,2) not null default 0.00 comment '支付金额，单位：元'
status tinyint not null default 1 comment '状态：1-启用，0-禁用'
```

## 表设计示例

以下示例表示历史用户名也不允许重复；如果允许软删后重建同名用户，需按上文逻辑删除场景调整唯一约束策略。

```sql
create table sys_user (
  id bigint not null comment '主键 ID',
  username varchar(64) not null comment '用户名',
  mobile varchar(20) null comment '手机号',
  email varchar(128) null comment '邮箱',
  status tinyint not null default 1 comment '状态：1-启用，0-禁用',
  create_time datetime not null comment '创建时间',
  update_time datetime not null comment '更新时间',
  create_by bigint null comment '创建人 ID',
  update_by bigint null comment '更新人 ID',
  deleted tinyint not null default 0 comment '逻辑删除标识：0-未删除，1-已删除',
  primary key (id),
  unique key uk_sys_user_username (username),
  key idx_sys_user_mobile (mobile),
  key idx_sys_user_status_create_time (status, create_time)
) comment = '用户表';
```

## SQL 编写规范

- 禁止拼接 SQL 参数，必须使用参数绑定。
- 禁止 `select *`，必须明确字段。
- 禁止无条件 `update`、`delete`。
- 大批量更新、删除必须分批执行。
- 分页查询必须带明确排序字段，避免分页结果不稳定。
- 排序字段必须白名单控制，不能直接使用前端传入字段拼接。
- 多表查询必须明确表别名。
- SQL 复杂时优先写清楚，不追求过度压缩。

## 数据库变更规范

数据库变更必须提交 SQL 脚本或迁移文件。

要求：

- 每次变更必须说明变更目的、影响范围和回滚方案。
- DDL 和 DML 分开提交。
- 生产变更前必须在测试环境验证。
- 大表 DDL 必须评估锁表、耗时和回滚风险。
- 新增字段应尽量设置默认值或允许空值，避免历史数据写入失败。
- 删除字段必须先确认无代码、报表、任务、接口依赖。
- 字段改名优先采用新增字段、同步数据、切换代码、删除旧字段的渐进方案。

## 迁移文件命名规范

如果项目使用 Flyway，推荐：

```text
V202606140001__create_sys_user_table.sql
V202606140002__add_mobile_index_to_sys_user.sql
```

如果项目未使用迁移工具，推荐：

```text
sql
├── ddl
│   └── 20260614_create_sys_user_table.sql
└── dml
    └── 20260614_init_sys_user_data.sql
```

## 数据初始化规范

- 初始化数据脚本必须可重复执行或明确执行条件。
- 初始化账号、密码、密钥不得使用生产真实数据。
- 初始化字典、菜单、权限必须有稳定编码。
- 初始化数据变更必须随代码一起评审。

## 性能规范

- 高频查询必须通过索引和执行计划验证。
- 大表查询必须带分页或范围条件。
- 避免在 where 条件中对索引字段使用函数。
- 避免隐式类型转换导致索引失效。
- 避免大事务长时间持锁。
- 避免一次性加载大量数据到内存。
- 报表、统计、搜索类需求应评估异步、缓存、宽表或搜索引擎方案。

## 安全规范

- 敏感字段必须评估加密、脱敏或访问控制。
- 密码不得明文存储。
- token、密钥、验证码不得落普通业务表明文。
- 身份证、手机号、银行卡等敏感数据应按合规要求存储和展示。
- SQL 日志不得输出敏感字段明文。
- 数据导出必须有权限控制和审计。

## 数据库评审清单

新增或修改表结构时检查：

- 表名、字段名是否小写下划线且语义清晰。
- 是否有主键。
- 是否包含必要基础字段。
- 字段类型和长度是否合理。
- 金额是否使用 `decimal`。
- 时间是否使用合适时间类型。
- 状态、类型字段是否有明确枚举值注释。
- 必填字段是否 `NOT NULL`。
- 默认值是否有业务含义。
- 是否需要唯一约束。
- 是否需要普通索引或联合索引。
- 联合索引字段顺序是否合理。
- 逻辑删除是否影响唯一约束。
- 是否存在大字段、JSON 字段或模糊查询风险。
- SQL 变更是否有回滚方案。
- 大表变更是否评估锁表和耗时风险。
- 是否涉及敏感数据和权限控制。
