本文件用于约束业务 API 的设计、变更和审查。具体 Spring Boot、FastAPI 实现细节仍需遵守对应技术栈规范。

## 适用范围

- 新增、修改、废弃或删除接口。
- 请求参数、响应字段、错误码、分页、排序、过滤。
- 文件上传、下载、导入、导出。
- 幂等、批量、异步任务、第三方回调接口。
- OpenAPI / Swagger 文档和接口兼容性评估。

## 基本原则

- API 必须稳定、清晰、可演进。
- 已发布接口优先保持兼容，破坏性变更必须说明影响和迁移方案。
- 接口设计应贴近业务资源，不为了实现细节暴露内部模型。
- 请求、响应、错误码和分页结构应在同一项目内保持一致。
- 特殊协议接口可按协议返回，但必须说明例外原因。

## 路径与方法

- 路径使用小写名词和中划线，资源集合使用复数名词。
- HTTP 方法语义必须清晰：`GET` 查询、`POST` 创建或提交动作、`PUT` 全量更新、`PATCH` 局部更新、`DELETE` 删除。
- 查询条件使用 query parameter，创建和复杂修改使用 JSON body。
- 不在路径中使用动词式接口名，如 `/getUser`、`/deleteUser`。
- 非 CRUD 动作应表达为资源状态或子资源，如 `/orders/{orderId}/cancel-requests`。

## 请求参数

- 请求参数必须有明确类型、必填规则、长度范围、枚举值和示例。
- 不信任前端传入的用户 ID、角色、租户、权限等敏感身份信息。
- 时间范围必须明确左右边界和时区。
- 金额字段必须明确单位和精度。
- 排序字段、导出字段、查询字段必须使用白名单。
- 请求 DTO / Schema 不直接复用数据库 Entity / ORM Model。

## 响应结构

- 业务 JSON 接口应使用项目统一响应结构。
- 响应 VO / Schema 不直接暴露 Entity / ORM Model。
- 响应字段命名应稳定，避免直接泄露内部表字段或第三方原始字段。
- 敏感字段、内部调试字段、权限表达式、堆栈和 SQL 不得返回前端。
- 空列表返回空数组，空对象和 `null` 的语义必须保持一致。

## 错误码与异常

- 可预期业务失败必须使用稳定错误码。
- 前端需要分支处理的错误必须有明确 code，不依赖文案判断。
- 同一类错误在不同接口中应保持相同 code 和相近 message。
- 参数错误、认证失败、权限不足、资源不存在、资源冲突应有清晰分类。
- 未知异常不得向调用方暴露堆栈、SQL、类名或内部路径。

## 分页排序过滤

- 页码、每页数量、总数、总页数和记录列表字段应统一。
- 页码默认从 1 开始，`pageSize` / `page_size` 必须设置最大值。
- 分页查询必须有稳定排序，避免翻页结果漂移。
- 排序字段必须白名单映射，不直接拼接前端输入。
- 复杂过滤条件应明确字段、操作符和组合关系，不让前端传任意 SQL 片段。

## 幂等与批量操作

- 创建订单、支付、回调、导入、消息消费等接口必须评估幂等。
- 可重复提交的写接口应使用业务唯一键、幂等号或状态机保证安全。
- 批量操作必须限制数量上限，并说明部分成功、全部回滚或失败明细策略。
- 批量删除、批量更新必须校验每个资源的权限和数据归属。
- 长耗时操作应考虑异步任务、任务 ID、进度查询和失败重试。

## 文件接口

- 文件上传必须限制大小、格式和业务类型。
- 文件下载、预览、导出必须校验权限。
- 导出接口应说明数据范围、字段范围和脱敏策略。
- 大文件或大批量导出应采用异步任务或流式响应。
- 文件名、路径和对象存储 key 不得直接信任用户输入。

## 版本与兼容性

- 对外接口必须有清晰版本策略，默认使用路径版本号或项目既有方案。
- 新增可选字段通常是兼容变更。
- 删除字段、修改字段类型、修改必填规则、改变枚举含义、改变响应层级属于破坏性变更。
- 破坏性变更必须升级版本或提供迁移期。
- 废弃接口应在文档中标记 deprecated，并说明替代方案。

## 文档要求

- 新增和修改接口必须同步更新 OpenAPI / Swagger 或项目接口文档。
- 文档必须包含 summary、参数说明、响应说明、错误码和示例。
- 特殊接口必须说明鉴权、幂等、限流、回调签名和兼容性。
- 文档不得包含真实密钥、生产地址、生产账号或敏感样例数据。

## API 交付清单

- 路径、方法和资源语义是否清晰。
- 请求参数是否有类型、校验、示例和业务说明。
- 响应是否使用统一结构并过滤敏感字段。
- 错误码是否稳定且可被前端处理。
- 分页、排序、过滤是否有上限和白名单。
- 写接口是否评估幂等、重复提交和事务边界。
- 是否涉及破坏性变更，是否需要升版本或兼容方案。
- 是否同步更新接口文档和测试。
